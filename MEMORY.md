### 3 次法则
> **当同一个操作失败 3 次时，必须停下来换方法**

- 失败 1 次 → 重试（可能是临时问题）
- 失败 2 次 → 检查参数/环境
- 失败 3 次 → 停止！换方法或求助用户

**真正的 resourceful 是懂得灵活切换策略，不是死磕**

### 5. 动态网页元素失效（2026-03-14）🔴
**问题**: GPT 页面发送消息时，snapshot 后等待太久导致 ref 失效，重复 10+ 次
**原因**: 
- React 应用持续重新渲染
- snapshot 只捕获瞬间 DOM

**解决**: 
- 创建强制执行系统（memory_guard.py）
- 失败计数器（3 次强制停止）
- 操作前强制检索记忆
- 操作后必须验证结果

### 19. 语义查询当前主要瓶颈不在向量数量，而在执行链（2026-03-19）🟠
**现象**:
- 结构化查询 `昨天做了什么` 约 45~56ms
- 语义查询 `browser automation dynamic page` 约 460ms 量级
- 即便把向量总数从 1554 压到 485，语义查询延迟仍几乎没有明显下降
**推断**:
- 当前主瓶颈更可能是 Python 进程启动、embedding/语义链路、rerank 等执行开销，而不是 FAISS 向量总量本身

### 24. 新规则生效后的 0 投，不一定是坏结果，而是“质量守门开始工作”的信号（2026-03-19）🟠

- 前程无忧 / 品牌策划 + 品牌专员：搜索 10 个，全部被严格岗位族/投递桶标准拦截，0 投
**意义**:
- 现在系统已经不会像以前那样“看到策划/高薪就投”
- 新规则开始真的守门，宁可 0 投也不脏投
**教训**: 0 投并不总是失败；如果召回结果本身不干净、边界模糊，0 投是比误投更健康的结果。真正需要后续优化的是岗位定义、召回词和岗位族规则，而不是把守门重新放松。

### 25. 浏览器咨询链路当前真正问题是“稳定交互层没打通”，不是单纯等待不足（2026-03-19）🔴
**问题**: 下午多次尝试去问 ChatGPT / Gemini 时，反复出现：刚 snapshot 拿到的输入框 ref 在 click/fill 前就失效；即使等待 10 秒后再 snapshot，Gemini 当前的 textbox 仍会在 fill 前失效。ChatGPT 那边则出现 `textarea.value` 改了但 UI/前端状态没有接受输入的情况。
**已确认**:
- 失败点不只是“没有等待页面稳定”
- 更关键的是：当前 external LLM 网页的 **真实输入层 / 提交层 / 回复流验证层** 还没有一条成功样本被稳定跑通
- 现在不能把 aria ref 当成唯一可靠锚点，也不能把 DOM value 变化当成真实写入成功
**正确原则**:
1. 先命中真实活输入框
2. 再验证 UI 可见输入
3. 再验证 prompt 回显
4. 再验证回复流
少任何一步都不能说“问到了”
**教训**: 先跑通一条真实可验证链路，再谈固定链路；不能先定方法再假设成功。
